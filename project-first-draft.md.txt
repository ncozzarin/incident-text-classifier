Project Definition: 
Public Security Report Anonymization & AI Classification System
1. Introduction
The "Sistema de Control Disciplinario de las Fuerzas de Seguridad" of Córdoba is a functionally independent body designed to investigate and sanction misconduct within the Police, Anti-narcotics Force (FPA), and Penitentiary Service. Given the sensitive nature of investigating the very forces meant to provide security, data integrity and privacy are paramount.
This software solution provides an anonymized processing layer for disciplinary files. It allows investigators to analyze patterns of institutional violence or corruption without exposing sensitive personal data of witnesses, victims, or officers until the formal judicial stage. The system then uses AI to classify the Type of Infraction and the Disciplinary Urgency to ensure that cases of high social impact (like Gatillo Fácil) are prioritized.
This project introduces a scalable, secure, and ethically grounded software solution. It serves a dual purpose:
1. Irreversible Anonymization: It strips sensitive data from reports using advanced, context-aware artificial intelligence and rule-based engineering while maintaining the grammatical integrity and semantic value of the text.
2. AI-Driven Triage: Once sanitized, it uses a supervised machine learning classifier to categorize the complaint and assess its urgency.
Originating in the province of Córdoba, Argentina, the system is designed to understand local security contexts but is architected to scale to other provinces, countries, and languages in the future.
________________


2. Core Goals and Needs
A. Data Privacy & Compliance
* The Need: Full adherence to data protection standards.
* The Goal: PII like names, IDs, emails, and phone numbers must be destroyed or replaced with tags (e.g., [PERSONA], [DNI]). Raw, un-anonymized data must be kept in a highly restricted, secure digital vault accessible only to specific authorized users for chain-of-custody and auditing purposes.
B. Triage (P1 to P5) and Classification
* The Need: Dispatchers and judicial officers are overwhelmed by the volume of incoming complaints.
* The Goal: The system will automatically execute a two-tier classification:
* Complaint Thopic: Assigning the complaint to specific legal and operational categories (such as Violencia de Género, Robo, Abuso de Autoridad). These will be governed and defined with examples by system administrators.
* Urgency Priority: Assigning a triage priority from P1 (Critical/Immediate Action) to P5 (Low Priority/Administrative).
* There should be a “Can’t define” option
C. Explainability & Human-in-the-Loop (HITL)
* The Need: Public security cannot rely on "black box" algorithms. Decisions must be transparent and defensible.
* The Goal: The system will display the AI’s confidence as a percentage breakdown. Furthermore, no automated classification is final; a human operator must validate and approve the AI’s suggested category and urgency level.
________________


3. Role-Based Access Control (RBAC)
To ensure strict governance, the system defines the following human personas:
* System Administrator: Manages users, defines categories, and provides the LLM with definitions and examples for classification.
* Data Operator: Responsible for uploading raw files and executing the bulk anonymization process.
* Prosecutor: Reviews the categorized incidents, views the P1–P5 urgency, and makes tactical or legal decisions.
* Researcher: Accesses fully anonymized datasets to pull statistics, map crime trends, and evaluate public security metrics without ever seeing PII.
3. Statistical and data analysis
* The Need: Having this kind of data could be useful to understand patterns, do statistics and find solutions
* The Goal: The system will see relationships between cases and expose them in natural language.

2. Anonymization Pipeline Sequence
To prevent rules from breaking each other, processing must strictly follow this order:
1. Emails: 100% extracted via standard syntax RegEx.
2. Account Numbers: Extracted via strict syntax RegEx.
3. Phone Numbers: Extracted via RegEx handling syntactic restrictions and regional prefixes.
4. DNI (Argentine ID): Extracted via isolated 7-to-8 digit boundaries.
5. Names & Surnames: First filtered via a compiled dictionary of common Argentine names (case-insensitive), followed by SpaCy's NER to catch edge cases.
6. Addresses: Handled mainly by Microsoft Presidio to safely detect street names and geographic markers.
7. Locations, Personas and all the rest is handled by spaCy
8. Tests should be added to the code to make sure no honeypots, false positives or false negatives are present
3. Architecture Approaches
* Approach A (Recommended for MVP): API-First Decoupled Architecture The React frontend is served independently. The Python 3.12 backend acts as a high-performance RESTful API .
   * Why: This allows the heavy AI compute loads to scale independently of the user interface.
* Approach B: Monolith in a Box Serving both React static builds and Python logic from a single server/container.
   * Why: Easiest to deploy on-premise inside secure police servers that lack external internet access.

Dual-Classification Integration Architecture
To handle both classifications simultaneously without doubling our server costs or latency, we have two primary architectural approaches for the model integration.
Approach A: The Multi-Task Learning (MTL) Pipeline
Instead of training two separate models, we use a single base model (like a Spanish BERT variant) with two separate output heads branching off the top layer.
* Head 1: Softmax output for the multi-class (or multi-label) Claim Type.
* Head 2: Softmax or regression output for the Urgency Priority (P1-P5).
Approach B: Decoupled Ensemble Pipeline
We run two completely independent models in parallel inside a worker pool.
* Model 1: Optimized for classification.
* Model 2: Trained specifically on sentiment, aggression, and urgency signals.

The Step-by-Step Data & Inference Flow
Here is exactly how data moves from the user's screen through the anonymizer and into the AI model:
1. Ingestion & Pre-processing
* The Data Operator uploads a CSV or the external API pushes a complaint text via the React frontend to the backend (Python 3.12).
* The payload goes into a message broker/queue (like Redis or RabbitMQ) to prevent blocking the web server during heavy AI inference.
2. Sanitization
* The worker pulls the raw complaint from the queue.
* It runs the multi-layered anonymization sequence we built earlier (Regex ➡️ Dictionary ➡️ SpaCy/Presidio).
* Crucial Step: The anonymizer outputs a clean string but retains a reference mapping to the raw text in our highly secure, encrypted database vault.

4. Database Persistence & WebSocket Push
* The system saves the predictions into the database as Unverified Drafts.
* The database structure stores: incident_id, anonymized_text, ai_suggested_crime, ai_suggested_urgency, ai_confidence_payload.
* A system admin corroborates the anonimization
* A WebSocket event pushes a notification to the React frontend telling the Prosecutor that a new report is ready for human review.
5. Human-in-the-Loop (HITL) Feedback Loop
* When the human validates or overrides the classification, that payload is saved in a "Gold Standard" training table.
* This table accumulates over time and will be pulled later to fine-tune and retrain the model.
